# Study Summarizer — Backend (C# / .NET 9)

A microservice that accepts documents (PDF/DOCX/TXT), extracts their text, and generates summaries.
A learning project built with C# + ASP.NET Core + EF Core + MySQL + Swagger, organized using
**Onion Architecture** (layered, with dependencies pointing inward toward the Domain).

## Architecture

The solution is split into 4 projects. Dependencies point **inward**: the Domain is the center
and knows nothing about the outer layers.

```
API  ──►  Service  ──►  Infrastructure  ──►  Domain
(HTTP)    (business)     (data access / EF)    (entities, center)
```

| Project | Responsibility | Depends on |
|---------|----------------|------------|
| **StudySummarizer.Domain** | Entities (`Document`, `Summary`, `BaseEntity`) and the `DocumentStatus` enum. Pure C#, no external dependencies. | — |
| **StudySummarizer.Infrastructure** | `AppDbContext` (EF Core) + EF migrations. Talks to MySQL. | Domain |
| **StudySummarizer.Service** | Business logic: `DocumentService`, `SummaryService`, text extraction, summarization. DTOs + interfaces + custom exceptions. | Domain, Infrastructure |
| **StudySummarizer.API** | Thin controllers, `Program.cs` (DI wiring), exception-handling middleware, Swagger. | Domain, Infrastructure, Service |

The controllers are **thin**: they only translate HTTP ↔ service calls. All real logic lives in the
Service layer. Services use `AppDbContext` directly (pragmatic onion — no generic repository, so EF
features like `.Include(...)` stay available).

## How to run the project

`dotnet` is installed in `~/.dotnet` and is NOT on the PATH. Before any command, run:

```bash
export DOTNET_ROOT="$HOME/.dotnet"
export PATH="$HOME/.dotnet:$PATH:$HOME/.dotnet/tools"
```
(or add these lines once to `~/.zshrc` to make them permanent)

Then start the API project:
```bash
dotnet run --project StudySummarizer.API --launch-profile http
```
Open Swagger: http://localhost:5138/swagger

## Database (MySQL)

- Server: `127.0.0.1:3306`, database: `studysummarizer`, user: `study`
- The connection string lives in **User Secrets** (not in the code!), tied to the API project. View it with:
  ```bash
  dotnet user-secrets list --project StudySummarizer.API
  ```
- Start MySQL (if stopped): `brew services start mysql`

### Migrations (when you change entities in Domain)
The DbContext lives in Infrastructure, but the EF tooling is started from the API project:
```bash
dotnet ef migrations add MigrationName \
  --project StudySummarizer.Infrastructure \
  --startup-project StudySummarizer.API

dotnet ef database update \
  --project StudySummarizer.Infrastructure \
  --startup-project StudySummarizer.API
```

## Project structure

```
StudySummarizer.sln
StudySummarizer.Domain/
  Entities/
    BaseEntity.cs       -> base class with the Id primary key
    Document.cs         -> Document entity + DocumentStatus enum
    Summary.cs          -> Summary entity
StudySummarizer.Infrastructure/
  Data/
    AppDbContext.cs     -> the "bridge" to the database
  Migrations/           -> schema change history (generated by EF)
StudySummarizer.Service/
  Dtos/
    Dtos.cs             -> API request/response objects + service inputs
  Interfaces/
    IDocumentService.cs, ISummaryService.cs
    ITextExtractor.cs, ISummarizer.cs
  Services/
    DocumentService.cs  -> upload, list, get, download, delete logic
    SummaryService.cs   -> summarize / get / regenerate logic
    TextExtractor.cs    -> extracts text from TXT/PDF/DOCX
    StubSummarizer.cs   -> FAKE summarization (to be replaced with real AI)
  Exceptions/
    ServiceExceptions.cs -> NotFoundException, ValidationException
StudySummarizer.API/
  Controllers/
    DocumentsController.cs, SummariesController.cs
  Middleware/
    ExceptionHandlingMiddleware.cs -> maps exceptions to HTTP status codes
  Program.cs            -> startup + DI + Swagger + CORS
```

## Endpoints

| Method | Route | Description |
|--------|-------|-------------|
| POST   | /api/documents | Upload a document (form-data, `file` field) |
| GET    | /api/documents | List documents |
| GET    | /api/documents/{id} | Document metadata |
| GET    | /api/documents/{id}/file | Download the original file |
| DELETE | /api/documents/{id} | Delete a document (+ its summary) |
| POST   | /api/documents/{id}/summarize | Generate a summary |
| GET    | /api/documents/{id}/summary | Read the summary |
| PATCH  | /api/documents/{id}/summary | Regenerate with different parameters |
